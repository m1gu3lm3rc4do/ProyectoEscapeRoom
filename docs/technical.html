Modelos de datos

GameSession

  • user FK → User
  • difficulty CharField (easy/medium/hard)
  • time_taken FloatField (segundos)
  • questions_answered IntegerField
  • correct_answers IntegerField
  • completed BooleanField
  • created_at DateTimeField (auto)

Room

  • title CharField (max 100)
  • description TextField (max 500)
  • difficulty CharField (easy/medium/hard)
  • status CharField (draft/published)
  • min_correct PositiveIntegerField
  • creator FK → User
  • created_at DateTimeField (auto)
  • published_at DateTimeField (nullable)

Question

  • room FK → Room
  • creator FK → User
  • text CharField (max 500)
  • option_0 … option_3 CharField (max 200)
  • correct_index PositiveSmallIntegerField (0–3)
  • created_at DateTimeField (auto)

RoomSession

  • room FK → Room
  • user FK → User
  • time_taken FloatField (segundos)
  • questions_answered IntegerField
  • correct_answers IntegerField
  • completed BooleanField
  • created_at DateTimeField (auto)

Autenticación JWT

Se usa djangorestframework-simplejwt. La configuración en settings.py:

SIMPLE_JWT = {
    "ACCESS_TOKEN_LIFETIME":  timedelta(hours=8),
    "REFRESH_TOKEN_LIFETIME": timedelta(days=7),
    "AUTH_HEADER_TYPES":      ("Bearer",),
}

El cliente envía el token en el header Authorization: Bearer <token>. El dashboard renueva el access token automáticamente usando el refresh token antes de que expire.

CORS_ALLOW_ALL_ORIGINS = True  # Desarrollo — restringir en producción

Frontend (Streamlit)

Archivo principal: Frontend/app.py

Gestión de estado

Streamlit es stateless por naturaleza. El estado de sesión se mantiene en st.session_state:

_SESSION_DEFAULTS = [
    ("token",          None),   # JWT access token
    ("refresh_token",  None),   # JWT refresh token
    ("user",           None),   # Nombre de usuario
    ("studio_room_id", None),   # Room seleccionada en Creator Studio
]

Renovación automática de token

La función api_request() envuelve todas las llamadas HTTP. Si recibe un 401, intenta renovar el access token con el refresh token antes de reintentar la petición.

Lanzamiento del Game Client

El juego se lanza como subproceso independiente con subprocess.Popen, pasando el token JWT y el nombre de usuario como argumentos de línea de comandos:

args = [venv_python, game_path, token, username, difficulty]
if room_id is not None:
    args.append(str(room_id))

subprocess.Popen(args, creationflags=CREATE_NEW_CONSOLE)

Game Client (pygame + PyOpenGL)

Archivos: game/main.py, game/api_client.py

Renderizado OpenGL

Se usa OpenGL legacy (modo inmediato) con glBegin/glEnd. La sala se renderiza en 3D y el HUD se superpone como una textura 2D con alpha blending.

ElementoTécnica
Sala (suelo, techo, paredes)Quads con glColor3f + glVertex3f
Puerta verde + marco doradoQuads coloreados en la pared del fondo (z = -6.9)
Personaje animadoQuads para cabeza, torso, brazos, piernas. Animación con math.sin(t * 6.0)
AntorchasTriángulos con interpolación de color naranja→amarillo usando math.sin(t * 8.0)
Baldosas del sueloGrid de quads con pequeño margen entre ellas
HUD 2DSurface pygame → textura RGBA → quad ortogonal con alpha blending

Pipeline HUD

# 1. Renderizar HUD en una Surface pygame con SRCALPHA
render_hud(hud_surf, fonts, state)

# 2. Subir la Surface como textura OpenGL RGBA
upload_hud_texture(tex_id, hud_surf)

# 3. Dibujar la textura sobre un quad ortogonal 2D
draw_hud_texture(tex_id)
# → glOrtho(0, WIDTH, 0, HEIGHT, -1, 1) + GL_BLEND

Animación del personaje

cycle     = t * 6.0  # frecuencia de ciclo de caminar
leg_swing = math.sin(cycle) * 0.28
arm_swing = -math.sin(cycle) * 0.22

El personaje se desplaza en el eje Z desde z = 2.5 (cerca de la cámara) hasta z = -5.5 (junto a la puerta). Cada respuesta correcta mueve el objetivo un paso hacia la puerta; cada fallo lo retrocede medio paso.

Antorchas — efecto de parpadeo

f = 0.5 + 0.5 * math.sin(t * 8.0 + phase)  # 0..1
g_c = 0.549 + 0.294 * f  # interpola naranja→amarillo

Cada antorcha tiene dos triángulos con fases distintas para un efecto más orgánico.

Decisiones de diseño

¿Por qué pygame + OpenGL en lugar de un motor 3D completo?

Un motor como Unity o Godot añadiría una dependencia externa enorme y requeriría un proceso de build separado. pygame + PyOpenGL permite distribuir el juego como un script Python puro, lanzable directamente desde Streamlit con subprocess.Popen, sin instaladores adicionales. El renderizado OpenGL legacy (glBegin/glEnd) es suficiente para la sala simple que necesita el juego y es fácil de entender y modificar.

¿Por qué Streamlit para el dashboard?

Streamlit permite construir una interfaz web interactiva con Python puro, sin necesidad de HTML/CSS/JS adicional. La gestión de estado con st.session_state es suficiente para el flujo de autenticación y navegación. Los gráficos de Plotly se integran de forma nativa. La alternativa (Django templates o React) habría requerido mucho más código de frontend.

¿Por qué SQLite?

SQLite elimina la necesidad de instalar y configurar un servidor de base de datos (PostgreSQL, MySQL). Para un proyecto de desarrollo/demostración con un único servidor, SQLite es más que suficiente. La migración a PostgreSQL en producción solo requiere cambiar DATABASES en settings.py.

¿Por qué JWT en lugar de sesiones de Django?

El game client (pygame) es un proceso independiente que no comparte cookies con el navegador. JWT permite pasar el token como argumento de línea de comandos al proceso del juego, que lo usa directamente en sus peticiones HTTP. Las sesiones de Django no funcionarían en este escenario.
